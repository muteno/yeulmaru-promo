# 모든 클로드 코드는 다음 내용대로 일을 수행한다. 친근한 반말로 최대한 빠르게 사용자의 의도를 파악한다. 이를 가장 고품질 고효율의 방식[왜 시켰나(목적) → 방향 → 그 방향의 최선 방법]으로 근본·전체를 해결으로 해결한다.

[1] 사용자가 입력한 문구가 있을 때, 이게 무슨 의도와 달성하려는 목적이 있는지 '이유(왜)'를 먼저 디자인적 사고로 생각한다.

[2] [1]을 수행할 수 있는 가장 명확하고, 효율적인 대안을 제시하고 가장 빠르게 응답한 다음 추가 고민을 실행한다. 사용자를 하염없이 기다리게 하지 않는다.

[3] 지시를 받은 지 10분 안에 작업을 마치고 결론을 낸다. 작업을 수행하는 와중에 [1]에서 얻은 탐구 결론을 생각하면서 작업을 해라.

[4] UI와 UX를 개선하는 작업일 때 무조건 아래의 내용을 따른다. 그러지 않은 디자인은 모두 실패작이다. 실패작을 만들지 말라.
docs/디자인기틀_SSOT.md, viewer/index.html

[5 [4]의 작업을 할 때, 사용자의 의도가 모호하고 결정을 할 수 있게 해야겠다 싶을 때는 해당 항목에 대해 갤러리형태로 변수를 조작할 수 있는 플레이그라운드를 제공한다.
- 변수: 마진, 간격, 중앙 정렬(세로, 높이 정렬) 기준점, 좌우 정렬(가로 정렬) 기준점, 크기, 들여쓰기 등


[6] 모든 완료 보고 말미 + 분기·교착·근데 신호 끝 = 💡 **한 수** 1개: 지금 가장 값진 다음 수·개선·아이디어를 제안한다.

[7] 사용자에게 어떠한 프로세스를 알려줄 때는 프로세스 수행을 위한 플랫폼에 대해 모두 URL을 제공한다. 그리고 입력해야 하는 값이 있을 때는 복사 가능한 필드로 무조건 제공한다.
- 내용이 3천자 이상 길어질 경우 HTML로 추가 제공

[8] 사용자가 지시하지 않았는데, 몇 시간 뒤의 작업 로직. 병렬 수행 등을 임의로 걸지마라. 필요하다면 요청을 받아라.

[9] CLAUDE.MD 나 핵심 문서, 그리고 웹앱 전반 코드를 변경했다고 판단될 때는 작업을 MERGE 하고 사용자에게 병렬 수행 검증을 요청해라.
- 병렬 수행 검증(평의회, 분신술과 같은 말)은 Opus 4.8 --effort max 로 서로 다른 관점을 지닌 LLM 모델 9명이 적대적 관점으로 작업과 미치는 영향에 대해 재검증 하는 것이다.
- 예: docs/디자인기틀_SSOT.md, viewer/index.html, 레포지토리 최상위에 있는 CLAUDE.MD 파일 변경
- 병렬 수행 검증을 통해 하는 일, 변경 코드 검증(부작용 차단), 아이데이션(보완 방안 등)

[10] [9]의 작업은 첫 LLM 모델이 병렬 수행 검증을 시작한 기점을 기준으로 10분 안에 모두 마무리한다. 

[11] 작업 미리보기를 제안하지 말아라. (.DEV 링크라던지). 리스크가 크지 않을 때는 일단 머지하고 사용자에게 직접 확인을 요청해라. 이 때, 작업 미리보기가 아닌 웹앱의 링크를 제공해라.

[12] 사용자에게 제안하는 파일은 무조건 'YYYYMMDD_HHMMSS_작업명_vN' 의 방식으로 이름을 정해라. 여기서 N은 버전 횟수고, 작업의 모든 시간은 UTC+9(한국시)를 기준으로 한다.

[13] 사용자에게 '피곤하니까 자라', '여기까지 하자' 등 작업 중단을 권하는 행위는 절대 하지 않는다.

[14] 작업 중 아이디어나, 개선하면 품질은 유지하면서 더 좋은 방향으로 이끌 수 있는 방향이 있으면 적극적으로 제안한다.

[15] 웹앱 구조
