# 🖥️ 대시보드 "브라우저에서 보이나?" — 조건·순서 (260724 실측 확립)

> **왜 이 문서?** "웹 브라우저에 대시보드 보이는지 봐줘"를 세션마다 다르게 답하던 문제(어떤 세션은 "안 된다", 어떤 세션은 스샷을 띄움)를 끝내려고, **되는 조건과 순서를 SSOT로 고정**한다. 새 세션은 이 문서부터 읽고 답한다.
>
> ⚠️ **비밀 금지**: 운영자 계정 이메일·앱 PIN·MS 비밀번호 등은 이 문서에 **절대 적지 않는다**(레포 = 공개 이력). 접속 값은 그때그때 운영자가 채팅으로 건넨다.

---

## 0. 한 줄 결론 (제일 먼저 이해할 것)

**"브라우저에서 대시보드를 본다"는 두 개의 서로 다른 일이고, 조건이 다르다.**

| 경로 | 무엇을 보나 | 어디서 되나 | 로그인 후 실데이터? |
|---|---|---|---|
| **A. 원격 세션 자체 렌더** | 앱을 **내가 새로 그린 복사본**(로그인 게이트까지) | claude.ai/code·웹·컨테이너 세션(=이 세션) | **불가** (아래 §2 이유) |
| **B. 진짜 탭 보기** | 운영자가 **이미 로그인한 실제 탭** | "클로드 인 크롬" 커넥터가 **붙은 표면**(브레이브 쪽) | 가능 (그 표면에서) |

→ **원격 코드 세션(A)은 로그인 게이트 화면까지만 그려서 보여줄 수 있다. 진짜 대시보드(로그인 후 실데이터)는 B에서만 보인다.** 이 둘을 섞으면 "된다/안 된다"가 세션마다 엇갈린다.

---

## 1. 경로 A — 원격 세션이 "앱 첫 화면"을 렌더하는 순서 (되는 방법)

이 세션(원격 컨테이너)에는 브라우저를 조종하는 커넥터가 없다. 대신 **헤드리스 Chromium(크롬과 같은 엔진)** 으로 `index.html`을 직접 띄워 스샷을 찍을 수 있다. 순서:

1. **의존성**: `npm install` (→ `playwright-core`). Chromium 바이너리는 원격 세션에 상주 = `/opt/pw-browsers/chromium-*/chrome-linux/chrome`. `playwright install` 금지(재다운로드 사고).
2. **리다이렉트 한 줄 제거**(치명 함정): `index.html` 최상단
   `<script>if(location.protocol==='http:')location.replace('https://'+...)</script>`
   때문에 `http://localhost`로 열면 **https로 튕겨 → 로컬은 https 미지원 → `ERR_SSL_PROTOCOL_ERROR`**. 렌더 시 이 줄만 제거한 사본을 띄우거나 `file://`로 연다(`file:` 프로토콜은 이 조건에 안 걸림).
3. **서빙**: 레포를 `127.0.0.1`(=`NO_PROXY`라 프록시 우회)로 로컬 서버(`python3 -m http.server 8765 --bind 127.0.0.1`)에 올리고 Chromium이 그 주소를 로드. 외부 리소스(구글폰트·jsDelivr·MSAL·Worker API)는 abort(→ §2 때문에 어차피 못 받음).
4. **결과**: `GS Caltex YEULMARU / Microsoft 계정으로 로그인` **로그인 카드가 정상 렌더**됨(데스크톱·모바일 모두). 화면 하단 빨간 `MSAL 라이브러리 로드 실패` 토스트는 **외부 CDN을 막아서 뜬 것 = 렌더 테스트의 정상 산물, 실 버그 아님**(라이브에선 안 뜸).

> 참고: 라이브 생존 확인은 브라우저 없이도 됨 — `curl -sS -o /dev/null -w '%{http_code}' https://promo.yeulmaru.org/` (프록시 경유, 200이면 살아있음).

## 2. 경로 A가 "로그인·실데이터"까지 못 가는 이유 (조건 미충족)

원격 컨테이너의 Chromium은 **외부 HTTPS에 못 닿는다**. 실측:

- 아웃바운드는 에이전트 프록시(`$HTTPS_PROXY` = `127.0.0.1:33979`, **CONNECT 터널만**)를 거쳐야 함. `curl`은 이 프록시로 외부에 잘 닿지만, **Chromium은 같은 프록시로 CONNECT가 `ERR_CONNECTION_RESET`** (예: `promo.yeulmaru.org`·`cdn.jsdelivr.net`·`example.com` 전부 리셋 — 프록시 릴레이 로그엔 Chromium의 평문 HTTP 프로브만 찍히고 CONNECT는 도달 안 함). = **샌드박스 브라우저의 환경 한계.**
- 따라서 (a) **MSAL(마이크로소프트 로그인) 라이브러리**를 CDN에서 못 받고, (b) **Worker API**(`yeulmaru-promo-api.*.workers.dev`)에 못 붙어 PIN 검증·데이터 로드가 안 됨.
- **로그인 흐름 자체가 MS SSO 선행**: `initLoginScreen()`(index.html) → **Microsoft 계정 로그인(MSAL)** 으로 이메일(`@gscf.or.kr`) 식별 → 앱 **PIN** → `_enterApp()`. **PIN만으로 MS 게이트를 건너뛰는 경로는 없다.** 즉 새 로그인은 Microsoft 대화형 인증(비번·MFA)이 필수라, 헤드리스로 대신 눌러줄 수도 없다.

**결론: 원격 코드 세션에서 "로그인해서 진짜 대시보드 보기"는 조건상 불가.** 가능한 건 §1의 로그인 전 화면 렌더까지.

## 3. 경로 B — 진짜 대시보드(로그인 후)를 Claude가 보게 하는 조건·순서

실데이터가 뜬 실제 화면은 **"클로드 인 크롬" 커넥터가 붙은 표면**에서만 볼 수 있다(운영자 환경 = 브레이브 + 클로드 인 크롬 확장 + 그 탭에 대시보드).

**조건(모두 충족돼야 함):**
1. 그 브라우저(브레이브 등)에 **"Claude for Chrome" 확장**이 설치·활성.
2. **같은 Claude 계정** 로그인.
3. **말 거는 그 대화(표면)에서** 해당 커넥터가 **켜져 있어야** 함(`enabledInChat=true`). ※ 이 원격 코드 세션엔 없음 — `ListConnectors`로 확인 가능.
4. 그 **탭 보기 권한**을 Claude에 허용(확장이 탭 단위로 물어봄).
5. 대시보드 탭이 **이미 로그인된 상태**(MS → PIN 완료)여서 실제 대시보드가 화면에 떠 있을 것.

**순서:** 브레이브에서 대시보드 열고 로그인(MS→PIN) → 그 탭에서 클로드 인 크롬 확장 활성 → 탭 접근 권한 허용 → **그 표면의 Claude에게** "이 탭 보고 ~해줘" 요청. (이 원격 세션에 아무리 시켜도 그 탭은 못 봄 — 커넥터가 여기 없어서.)

---

## 4. 새 세션 응답 규약 (요약)

"브라우저에 대시보드 보이나?" 를 받으면:
- **먼저 구분**해 답한다: 로그인 **전 화면 렌더**(경로 A, 이 세션서 가능) vs **실제 탭/실데이터**(경로 B, 커넥터 표면서만).
- 경로 A로 보여줄 땐 §1 순서대로 렌더 + 스샷. **MSAL 토스트는 정상 산물이라고 명시**(실 버그로 오해 금지).
- 실데이터를 원하면 §3 조건을 안내(못 하는 걸 "됐다"고 하지 말 것). 라이브 생존만 물으면 `curl` 200 확인으로 충분.
